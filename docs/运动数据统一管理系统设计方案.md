# 运动数据统一管理系统设计方案
> 版本：v1.0 | 更新时间：2026-03-12 | 适用场景：个人多品牌运动设备数据统一管理

---

## 一、总体架构设计
采用分层架构设计，从下到上共7层，兼顾数据可靠性、扩展性和易用性：

### 1.1 基础层（新增补充）
| 层级 | 核心能力 |
|------|----------|
| 原始数据备份层 | 所有导入的原始文件（ZIP/FIT/GPX/JSON）原样存储，永不删除，支持归一化逻辑出错时回溯重算、新增字段时重新解析旧数据 |
| 数据血缘层 | 记录每一条数据的来源、加工链路，标记每个指标的原始数据源，异常排查时可快速定位问题 |

### 1.2 核心业务层（原有优化版）
| 层级 | 核心能力 | 优化点补充 |
|------|----------|------------|
| 数据接入层 | 适配多品牌设备格式：<br>• 华为：解析Health app导出的ZIP包/JSON格式<br>• 高驰：解析FIT/GPX/TCX标准格式 + 高驰开放API对接<br>• 扩展预留：佳明、苹果健康等其他品牌接口 | 新增幂等性校验：导入前做文件哈希校验，重复文件自动跳过；损坏文件自动归入「导入失败」队列，记录错误原因，不阻塞批量导入 |
| 数据归一化层 | 定义统一标准数据模型（参考FIT标准扩展）：<br>• 基础信息：运动类型、开始/结束时间、总时长、距离<br>• 运动指标：配速、心率、步频、步幅、爬升、卡路里<br>• 轨迹数据：GPS点序列（时间/经纬度/海拔/速度）<br>• 生理数据：训练负荷、恢复时间、VO2Max估算<br>• 设备元数据：记录设备、数据源标记 | 采用Pydantic做类型校验，避免脏数据入库 |
| 数据处理层 | • 智能去重：通过时间+GPS特征识别同一次运动的多设备记录<br>• 冲突解决：多设备数据冲突时，智能合并更准确的读数<br>• 数据校验：异常值检测、补全缺失字段 | 新增异步队列：大文件/批量导入后台异步执行，前端实时展示进度，避免页面卡死；GPS轨迹采用Polyline算法压缩，存储体积减少70% |
| 业务逻辑层 | • 统一数据仓库：全量数据持久化存储<br>• 多维度查询：按时间、运动类型、设备等维度筛选<br>• 数据分析：跑量统计、PB追踪、趋势分析、训练负荷计算<br>• 数据导出：按品牌格式要求生成对应导出文件 | 常用统计结果做缓存，查询速度不随数据量增长下降；支持多条件组合空间查询（如「找出所有经过香山的跑步记录」） |
| 展示/交互层 | • Web管理面板：数据总览、详细记录查看、统计图表<br>• 操作界面：手动导入/导出、数据编辑、冲突解决<br>• 自动化：自动检测新导入文件、定时备份 | 支持离线地图加载轨迹，无需申请第三方地图API密钥；支持Telegram/微信端命令式操作，无需打开Web页面 |

## 二、技术选型（个人开发最优栈）
| 层级 | 技术选择 | 优势 | 优化补充 |
| ------- | ------------------------------------------- | --------------------------------------- | ---------- |
| 后端 | Python + FastAPI | 开发速度快、生态完善，运动数据处理库丰富 | 新增pydantic-settings做配置管理，多环境无缝切换 |
| 数据库 | SQLite（本地使用）/ PostgreSQL（部署用） | SQLite零配置，足够个人使用；可无缝升级到PostgreSQL | PostgreSQL默认开启PostGIS扩展，GPS轨迹空间查询效率提升100倍+，原生支持轨迹计算、区域匹配能力 |
| 前端 | 方案1：Streamlit（快速原型）<br>方案2：React/Vue（定制化界面） | Streamlit无需前端经验，1天就能做出可用界面；React适合做复杂交互 | 地图组件选Leaflet/MapLibre开源离线方案，支持离线加载轨迹 |
| 数据处理 | fitparse（解析FIT）、gpxpy（解析GPX）、pandas（数据分析） | 成熟的运动数据处理库，无需从零写解析逻辑 | 新增polyline库做轨迹压缩，自定义抽象适配层解决多品牌格式兼容问题 |
| 可视化 | Plotly/Matplotlib | 生成专业的运动数据图表，满足数据控需求 | 支持导出静态图表/交互式HTML报告 |
| 移动端（可选） | Flutter/PWA | 跨端开发，一次编写多端运行 | PWA支持离线访问，无需安装APP |
| 任务队列 | RQ + Redis | 轻量易用，适合个人项目异步场景 | 无需上重型Celery，满足批量导入、自动备份等异步需求 |
| 数据安全 | AES加密 + 压缩备份 | 敏感健康数据加密存储，备份文件自动加密压缩 | 支持双冗余备份：本地备份 + 自动上传个人OSS/云盘 |

## 三、核心功能规划
### 3.1 基础功能（MVP优先实现）
1. 多源导入：
 • 支持华为运动健康ZIP包批量导入
 • 支持高驰FIT/GPX文件导入
 • 自动识别文件类型，一键导入，重复文件自动跳过
 • 导入失败文件自动归档，记录错误原因
2. 数据统一管理：
 • 所有运动数据统一列表展示，标记来源设备
 • 详情页展示完整运动指标、轨迹地图
 • 支持手动编辑、删除错误数据
 • 每条数据可追溯原始文件和加工链路
3. 数据导出：
 • 通用格式导出：FIT/GPX/TCX
 • 品牌专属格式导出：分别生成华为/高驰可直接导回设备的格式
 • 批量导出选择时间段的数据
 • 支持脱敏导出，分享时隐藏隐私指标

### 3.2 进阶功能（数据控专属）
4. 智能去重合并：
 • 自动识别同一次运动的多设备记录：<br>「有GPS场景」：开始时间±5分钟 + 运动类型 + GPS轨迹首末点匹配<br>「无GPS室内场景」：开始时间±5分钟 + 运动类型 + 总时长误差<5% + 总卡路里误差<10%
 • 提供合并建议，支持用户自定义默认优先数据源（如默认优先高驰心率、优先华为GPS）
 • 支持数据补全：缺失字段自动从其他同次运动记录中补全，生成更完整准确的单一运动记录
 • GPS漂移、心率跳点自动平滑，支持手动回滚
5. 高级数据分析：
 • 多维度统计：月度/年度跑量、配速趋势、PB历史
 • 交叉对比：不同设备的数据准确性对比分析
 • 训练负荷分析：过载提醒、恢复建议
 • 自定义报表：按需求导出统计报告
 • 空间查询：按轨迹区域、爬升区间等维度筛选运动记录
6. 自动化能力：
 • 监控多个指定文件夹（iCloud下载目录、高驰同步目录等），新文件自动识别导入
 • 自动备份全量数据，支持版本回滚、双冗余备份
 • 对接OpenClaw实现每周自动生成运动报告，推送至指定聊天渠道
 • 支持命令式操作：Telegram/微信直接发送命令导出数据、生成报告

## 四、重点难点解决方案
1. **华为格式解析**：参考开源项目Hitrava的实现代码，做一层抽象适配层，华为更新APP导出格式时仅需修改适配层，无需改动核心逻辑；建立格式测试用例库，收集不同版本导出文件，每次升级自动跑兼容性测试。
2. **数据去重算法**：混合时间+特征+空间判断逻辑，准确率95%以上；支持用户手动标记重复记录，优化算法准确率。
3. **导出设备兼容性**：严格按照各品牌官方格式规范生成文件，导出前先生成1条测试数据导回设备验证可用性，再批量导出；建立导出兼容性测试用例库，避免regression。
4. **数据准确性校验**：导入导出时做MD5校验，原始文件永久备份，所有数据加工操作留痕可追溯；敏感数据加密存储，避免泄露。
5. **性能优化**：数据量超过1万条时自动开启按年月分区分表，常用统计结果缓存，查询速度不受数据量增长影响。

## 五、开发路线图
| 阶段 | 周期 | 核心任务 | 卡点提前规避 |
|-------|---------|------------------|------------------|
| Phase 1 | 1~2周 | 最小可用版本：<br>1. 原始数据备份、幂等校验、错误日志基础能力开发<br>2. 华为ZIP和高驰FIT文件解析<br>3. 统一数据模型定义，数据持久化存储<br>4. 简单列表展示和详情查看功能 | 先做基础可靠性能力，不要先写解析逻辑，避免后期数据丢失或出错无法回溯 |
| Phase 2 | 2~3周 | 核心功能完成：<br>1. 智能去重合并功能<br>2. 基础统计分析和图表展示<br>3. 通用格式导出（FIT/GPX/TCX）<br>4. 品牌专属格式导出基础能力 | 先做通用格式导出，再做品牌专属导出，通用格式可满足90%导出需求 |
| Phase 3 | 3~4周 | 体验优化：<br>1. Web管理界面完善<br>2. 自动化导入和备份功能<br>3. 高级数据分析模块<br>4. 一键数据迁移工具（SQLite→PostgreSQL） | 提前做数据迁移工具，后续部署到云端无需手动导数据 |
| Phase 4 | 可选 | 扩展能力：<br>1. 对接华为/高驰开放API实现自动同步<br>2. 移动端PWA/APP开发<br>3. 云端部署支持多设备访问<br>4. OpenClaw Skill开发，支持命令式操作 | 数据库设计预留user_id字段，后续支持多租户/云服务无需改表结构 |
